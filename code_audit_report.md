# 🔍 Аудит кода los_terminator.py

## Найденные баги (КРИТИЧЕСКИЕ)

### 1. 🐛 Telegram-сообщение врёт для новой аварии
**Строка 1242** — при НОВОЙ массовой аварии (Сценарий Б) отправляется текст:
> *«Заявка взята в работу. Полный отчёт в комментариях.»*

Но мы **больше НЕ берём заявку в работу** (по просьбе пользователя). Текст надо исправить.

---

### 2. 🐛 Старый `port_counter` конфликтует с новой OLT-логикой
**Строки 1373-1380** — Старый механизм `MASS_OUTAGE_THRESHOLD = 3`:
```python
port_counter[port_key] += 1
if port_counter[port_key] >= MASS_OUTAGE_THRESHOLD:
    log.warning(...)
    processed_ids.add(ticket_id)
    return olt_data  # ← МОЛЧА ПРОПУСКАЕТ! Ни комментария, ни TG, ничего!
```

**Проблема:** Если в одном цикле 3+ заявки с одного порта, этот блок молча пропустит 3-ю заявку — без проверки ЛОС, без отчёта, без Telegram. Он просто `return olt_data` и всё. А новый механизм (через OLT, строки 1487-1497) сделает ту же проверку правильно.

**Решение:** Удалить этот устаревший блок.

---

### 3. 🐛 Переменные `detail_output` и `inactive_output_clean` не инициализированы
**Строки 903-940** — Если OLT отвечает на `show gpon inactive-onu`, но падает до/во время `show gpon onu-description`, переменная `detail_output` не будет определена, и код упадёт с `NameError` на строке 947.

**Решение:** Инициализировать обе переменные перед `try`.

---

## Замечания по логике (СРЕДНИЕ)

### 4. ⚠️ Двойное подключение к OLT
Для каждой заявки с ЛОС скрипт подключается к OLT **дважды**:
1. `connect_olt()` (строка 1383) — basic-info + optical
2. `check_mass_outage_on_olt()` (строка 1487) — inactive-onu + onu-description

Это 2 отдельные telnet-сессии к одному и тому же устройству. На медленных OLT это лишние 10-15 секунд.

> [!NOTE]
> Это не баг, а оптимизация на будущее. Можно объединить в одну сессию позже.

---

### 5. ⚠️ `find_matching_emergency` делает API-запрос на КАЖДУЮ открытую аварию
Если открыто 20 аварий, скрипт делает 20 GET-запросов к `/ticket-comments`. При 50 авариях — 50 запросов.

> [!NOTE]
> Пока аварий мало, это не критично. Если станет тормозить — добавить кеширование.

---

### 6. ⚠️ `MASS_OUTAGE_THRESHOLD` (=3) vs `MASS_OUTAGE_MIN_CLIENTS` (=4)
Две переменные для одной задачи. `MASS_OUTAGE_THRESHOLD` использовался старым `port_counter`, а `MASS_OUTAGE_MIN_CLIENTS` — новым OLT-механизмом. После удаления `port_counter` первый становится ненужным.
