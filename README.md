# GPON Auto-Dispatcher (los_terminator)

Автоматизированный бот-диспетчер (NetDevOps) для диагностики оптических линий GPON и маршрутизации инцидентов. 

Проект разработан для автоматизации работы инженеров технической поддержки 2-й линии. Скрипт в фоновом режиме опрашивает API тикет-системы, подключается к коммутаторам (OLT), собирает физические параметры портов, проверяет наличие массовых аварий на сплиттере/OLT и на основе заложенной логики распределяет одиночные инциденты на выездные бригады (ВОЛС) либо привязывает к существующим авариям и переводит/возвращает их на первую линию (L1).

---

## 🚀 Основные возможности

1. **Интеграция с HelpDesk (API):**
   * Автоматический сбор открытых инцидентов через REST API (GET-запросы с токенизированной авторизацией).
   * Автоматическое взятие заявки в работу, добавление подробных HTML-комментариев с диагностикой и смена исполнителя (POST-запросы).
2. **Аппаратная диагностика коммутаторов (OLT):**
   * Подключение по SSH/Telnet с использованием библиотеки `netmiko`.
   * Автоматическое отключение пагинации CLI (`terminal length 0` / `page-control disable`), чтобы гарантировать полное чтение логов.
   * Парсинг CLI-вывода (мощность сигнала Rx/Tx, температура, напряжение, токи смещения, история деактивации ONU) с помощью регулярных выражений (`re` / RegEx).
3. **Умные сценарии обработки инцидентов:**
   * **LOS (Loss of Signal):** Детектирование отсутствия оптического линка на основе истории аварий `Losi`.
   * **Weak Signal (Высокое затухание):** Анализ уровней сигнала с градацией: критический сигнал (хуже `-32.0 dBm`), слабый в обе стороны (хуже `-28.5 dBm`) или сильное затухание на стороне клиента (хуже `-30.0 dBm`).
   * **Flap Detection (Флапающий линк):** Обнаружение нестабильных линков (>= 5 падений за последние 24 часа), даже если оптика временно недоступна (состояние Mib Syncing).
   * **Anti-Storm (Защита от массовых аварий):** Агрегация заявок по PON-портам. Если на одном порту падает >= 3 клиентов, бот автоматически распознаёт это как массовую аварию (даже если её ещё нет в общем списке), блокирует создание дублирующих заявок на выезд ВОЛС, формирует подробный HTML-отчёт с диагностикой аварии (количество клиентов на порту, статус, мощность) и вставляет его в комментарий к заявке, после чего возвращает её на 1 линию (L1) и сигнализирует в Telegram.
   * **Smart Routing (Умная маршрутизация):** Заявки с указанием зоны "Конаев" автоматически маршрутизируются на отдельную группу ("Филиал г.Конаев", исполнитель Алексей), остальные заявки уходят на общую группу ВОЛС.
4. **Уведомления:**
   * Отправка мгновенных алертов о переведенных на ВОЛС заявках в Telegram-канал дежурной смены.
5. **Daemon Mode (Режим демона):**
   * Непрерывная фоновая работа с кэшированием обработанных тикетов в `Set` (защита от повторной обработки).

## 🛠 Стек технологий

* **Язык:** Python 3.10+
* **Работа с сетью и API:** `requests`, `urllib3`
* **Управление сетевым оборудованием:** `netmiko` (SSH/Telnet)
* **Парсинг логов:** Regular Expressions (`re`)
* **Безопасность:** `python-dotenv` (хранение секретов вне кода)
* **Уведомления:** Telegram Bot API
* **Логирование:** библиотека `logging` (запись логов в `los_terminator.log`)

---

## 📁 Структура проекта

* `los_terminator.py` — основной боевой скрипт (содержит логику обработки, подключений и API).
* `test_ticket.py` — утилита для изолированного тестирования конкретного тикета по его ID без запуска общего цикла.
* `requirements.txt` — список необходимых Python-библиотек для установки.
* `.env` — файл конфигурации с секретными доступами и токенами (не загружается в репозиторий).
* `.env.example` — шаблон конфигурационного файла.
* `.gitignore` — исключение кэша, логов и конфигов из репозитория.

## ⚙️ Установка и запуск

### Вариант 1: Запуск через Docker (Рекомендуемый для продакшена)
Самый быстрый и надежный способ развернуть бота в фоне (с автоперезапуском):

1. Клонируйте репозиторий:
   ```bash
   git clone <URL_вашего_репозитория>
   cd los-terminator
   ```
2. Создайте файл `.env` на основе `.env.example` и заполните свои доступы:
   ```bash
   cp .env.example .env
   nano .env
   ```
3. Соберите и запустите контейнер:
   ```bash
   docker compose up -d --build
   ```
*Для просмотра логов в реальном времени используйте команду `docker compose logs -f`*

### Вариант 2: Запуск через venv (Для локальной разработки)

1. Клонируйте репозиторий и перейдите в папку проекта.
2. Создайте и активируйте виртуальное окружение:
   ```bash
   python3 -m venv venv
   source venv/bin/activate  # Для Linux/WSL
   # или venv\Scripts\activate для Windows
   ```
3. Установите зависимости:
   ```bash
   pip install -r requirements.txt
   ```
4. Создайте файл `.env` на основе `.env.example` и заполните свои доступы.
5. Запустите скрипт:
   ```bash
   python los_terminator.py --loop
   ```
