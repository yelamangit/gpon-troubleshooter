## ПРОБЛЕМА
В `los_terminator.py` (строки 975-1007, Сценарий А: КЛИЕНТ ОФЛАЙН) бот при обнаружении
свежего ЛОС **сразу отправляет заявку на ВОЛС** через `dispatch_to_vols()` без проверки
того, является ли это частью массовой аварии на порту OLT.

Существующая «защита» через `port_counter` (строки 881-885) — это всего лишь счётчик
тикетов **в текущем цикле обработки** (не проверяет OLT напрямую). Если тикеты приходят
в разных циклах, счётчик обнуляется.

## РЕШЕНИЕ
Портировать логику из SecureCRT скрипта `core.py → check_if_client_down()` в los_terminator.

### Новый алгоритм (Сценарий А):
```
КЛИЕНТ ОФЛАЙН + СВЕЖИЙ ЛОС →
  1. Телнет к OLT: sh gp ina | include <дата_падения>
  2. Парсить все упавшие клиенты + время
  3. Считать клиентов в окне ±3 мин от last_losi_time
  4. ЕСЛИ >= 4 клиентов → МАССОВАЯ АВАРИЯ:
     a) Проверить emergency_cache (уже открытые аварии)
     b) Если авария уже есть → комментарий с ссылкой + вернуть на L1
     c) Если новая → взять в работу + комментарий с полным отчётом + TG уведомление
  5. ЕСЛИ < 4 → одиночный ЛОС → на ВОЛС (как раньше)
```

### Параметры:
- Временное окно: ±3 минуты (для Los/Losi/Tiwi)
- Порог массовой: 4 клиента (3 доп. + 1 текущий)
- При массовой (новая): взять в работу → комментарий с полным отчётом → TG
- При массовой (существующая): комментарий с номером аварии → вернуть на L1
- L1_DEPT_ID: нужно уточнить у пользователя (задать в .env)

### Формат отчёта (взят из reporting.py):
```
ОТЧЕТ ОБ АНАЛИЗЕ ОТКЛЮЧЕНИЙ
Дата: 2026-07-12 15:30:00
Узел: hostname (olt_ip)
Порт: GPONport
Клиентов упало: X

Порт          | Описание         | S/N              | Статус    | Время падения       | Причина
GPON0/8:1     | Иванов_И.И.      | HWTC:AB12CD34    | off-line  | 2026-07-12 15:30:12 | Losi
GPON0/8:5     | Петров_П.П.      | HWTC:EF56GH78    | off-line  | 2026-07-12 15:30:14 | Losi
```
