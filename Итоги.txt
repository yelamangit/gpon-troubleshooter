1) Я разработал умного бота-диспетчера для нашей интернет-сети (GPON). Раньше, когда у клиента пропадал интернет, инженерам 2-й линии приходилось вручную открывать каждую заявку, долго подключаться к коммутатору, проверять сигнал и писать отчет для выездной бригады. Мой скрипт делает это сам за несколько секунд: он мониторит тикетницу, сам логинится на оборудование, проверяет физическое состояние линии (например, обрыв кабеля или плохой сигнал), пишет подробный отчет со всеми показателями и сразу передает заявку монтажникам. Это полностью избавило инженеров от рутины, предотвращает отправку десятка людей на одну массовую аварию и сильно ускоряет выезд монтажников к клиенту

2) Я разработал автоматизированный NetDevOps-пайплайн на Python для диагностики GPON-сетей. Скрипт интегрирован с нашей тикет-системой через REST API. Он работает как демон: периодически забирает новые заявки, парсит данные клиента (IP коммутатора, порты) и через библиотеку netmiko устанавливает Telnet/SSH сессии с OLT-коммутаторами. Скрипт собирает физические параметры линии (затухание, напряжение) и историю отключений, а затем пропускает их через логические сценарии:

Анализирует уровни сигнала RxPower в обе стороны, выявляя аномалии.
Ищет свежие события физического обрыва кабеля (Losi).
Распознает "флапающие" линки (постоянные обрывы). Если проблема подтверждается аппаратно, бот через API переводит тикет в работу, формирует HTML-отчет с диагностикой и отправляет его на выездную бригаду (ВОЛС), параллельно уведомляя команду через Telegram. Также реализована защита от массовых аварий — бот агрегирует заявки по портам и стопает дублирующие выезды.»

3) Уровень 3: Хардкорно-технический (Для лида / Senior-инженеров)
«Я спроектировал отказоустойчивый авто-диспетчер на Python для превентивного мониторинга OLT. Архитектура и стек: Python, REST API (requests), Netmiko, Regex, Telegram API. Ключевой функционал:

CLI Парсинг: Написал сложные регулярные выражения для извлечения табличных данных с OLT (команды basic-info, optical-transceiver-diagnosis), получая уровни Rx/Tx, токи смещения, температуру и историю падений (Deactive Reason).
Smart-маршрутизация: Реализована конечная машина состояний. Если абонент OFFLINE — анализируется таблица disconnect_history (валидация свежести Losi с учетом таймстемпов). Если ONLINE — проверяются пороговые значения затуханий с разделением логики (от клиента к узлу и наоборот). Внедрил алгоритм обнаружения "Flapping" линков (>= 5 обрывов за 24ч при отсутствии оптики в статусе Mib Syncing).
Стейт-менеджмент и оптимизация: Скрипт использует in-memory Set для кеширования ID обработанных тикетов для работы в режиме loop-демона. Реализован механизм агрегации (Port Counter) на базе defaultdict, выявляющий паттерны массовых аварий (>= 3 заявок на PON-порт).
Terminal Control: Обошёл ограничение CLI-оборудования на количество выводимых строк (пагинация --More--) с помощью инъекции ter
